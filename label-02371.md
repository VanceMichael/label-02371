## 第五次对话 - Bug修复

### 用户需求

1. 用户端的用户注册的手机号填写要进行数据填写验证
2. 用户端的注册接口返回 500 错误
3. 用户端登录接口返回 500 错误
4. 管理后台登录接口返回 500 错误

### 问题分析

500 错误的根本原因是 Hutool 的 `BCrypt` 类依赖 BouncyCastle 加密库，但项目中未引入该依赖，导致密码加密/验证时抛出异常。

### 解决方案

#### 1. 修复密码加密问题

**pom.xml** - 添加 Spring Security Crypto 依赖：
```xml
<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-crypto</artifactId>
</dependency>
```

**PasswordUtil.java** - 改用 Spring Security 的 BCryptPasswordEncoder：
```java
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

public class PasswordUtil {
    private static final BCryptPasswordEncoder ENCODER = new BCryptPasswordEncoder();
    
    public static String encode(String password) {
        return ENCODER.encode(password);
    }

    public static boolean matches(String rawPassword, String encodedPassword) {
        return ENCODER.matches(rawPassword, encodedPassword);
    }
}
```

**schema.sql** - 更新初始化用户密码哈希值为 Spring Security BCrypt 格式

#### 2. 手机号验证

**Register.vue** - 添加手机号格式验证：
- 正则表达式：`/^1[3-9]\d{9}$/`
- 失焦时验证
- 错误提示样式
- 提交前校验

## 第五次对话（续2）- 修复密码哈希问题

### 问题

使用 README 中的账号密码登录提示"用户名或密码错误"

### 原因

数据库初始化脚本中的 BCrypt 哈希值与实际 `123456` 加密后的值不匹配

### 解决方案

创建 `DataInitializer.java`，在应用启动时自动更新所有用户密码为正确的哈希值：

```java
@Component
public class DataInitializer implements CommandLineRunner {
    @Override
```

### 用户需求

### 问题分析

注册时用户尚未登录，`UserContext.getUser()` 为 null，导致操作日志中没有记录操作人，前端显示为"匿名用户"。

### 解决方案

```java
if (UserContext.getUser() != null) {
    operationLog.setUserId(UserContext.getUserId());
    operationLog.setUsername(UserContext.getUsername());
} else {
    // 对于注册操作，从参数中提取用户名
    if ("用户注册".equals(logAnnotation.operation())) {
        String[] paramNames = signature.getParameterNames();
        Object[] args = point.getArgs();
        for (int i = 0; i < paramNames.length; i++) {
            if (args[i] != null && args[i].getClass().getSimpleName().contains("RegisterRequest")) {
                try {
                    Object usernameValue = args[i].getClass().getMethod("getUsername").invoke(args[i]);
                    if (usernameValue != null) {
                        operationLog.setUsername(usernameValue.toString());
                    }
                } catch (Exception e) {
                    log.debug("获取注册用户名失败", e);
                }
                break;
            }
        }
    }
}
```

```java
/**
 * 状态流转规则：当前状态 -> 允许的目标状态集合
 */
private static final Map<Integer, Set<Integer>> STATUS_TRANSITIONS = new HashMap<>();
static {
    // 待确认 -> 已确认、已取消
    STATUS_TRANSITIONS.put(STATUS_PENDING, Set.of(STATUS_CONFIRMED, STATUS_CANCELLED));
    // 已确认 -> 已入住、已取消
    STATUS_TRANSITIONS.put(STATUS_CONFIRMED, Set.of(STATUS_CHECKED_IN, STATUS_CANCELLED));
    // 已入住 -> 已完成
    STATUS_TRANSITIONS.put(STATUS_CHECKED_IN, Set.of(STATUS_COMPLETED));
    // 已完成/已取消 -> 无法变更
    STATUS_TRANSITIONS.put(STATUS_COMPLETED, Set.of());
    STATUS_TRANSITIONS.put(STATUS_CANCELLED, Set.of());
}

private void validateStatusTransition(Integer currentStatus, Integer newStatus) {
    Set<Integer> allowedStatuses = STATUS_TRANSITIONS.get(currentStatus);
    if (!allowedStatuses.contains(newStatus)) {
        throw new BusinessException("状态流转不合法：不能从「当前状态」变更为「目标状态」");
    }
}
```

#### 2. LogAspect - 标准 JSON 序列化

**问题**：手动拼接字符串生成的参数格式不是严格 JSON，特殊字符（引号、换行等）未转义，影响审计检索。

**修复**：引入专用 Jackson ObjectMapper，先将参数转为 `Map<String, Object>` 过滤敏感字段，再由 Jackson 序列化为标准 JSON：

```java
private static final ObjectMapper LOG_MAPPER = createLogMapper();

private static ObjectMapper createLogMapper() {
    ObjectMapper mapper = new ObjectMapper();
    mapper.registerModule(new JavaTimeModule());
    mapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
    mapper.disable(SerializationFeature.FAIL_ON_EMPTY_BEANS);
    mapper.setSerializationInclusion(JsonInclude.Include.NON_NULL);
    return mapper;
}

private String buildSafeParams(String[] paramNames, Object[] args) {
    Map<String, Object> paramMap = new LinkedHashMap<>();
    for (int i = 0; i < paramNames.length; i++) {
        if (isSensitiveField(paramNames[i])) {
            paramMap.put(paramNames[i], "***");
        } else if (arg == null) {
            paramMap.put(paramNames[i], null);
        } else if (isPrimitiveOrWrapper(arg.getClass())) {
            paramMap.put(paramNames[i], arg);
        } else {
            paramMap.put(paramNames[i], sanitizeObject(arg));
        }
    }
    return LOG_MAPPER.writeValueAsString(paramMap);
}
```

改进点：
- 输出为标准 JSON，特殊字符自动转义
- 日期类型正确格式化（JavaTimeModule）
- 敏感字段仍然脱敏过滤
- 与 Spring 全局 ObjectMapper 隔离，互不影响

### 用户问题

1. 管理后台预订管理切换预订状态接口返回 500 错误
2. 用户端日期选择器：打开后今天视觉上选中但点确定无效；选择 3.5 日但显示选中的是 3.4 日

### 问题分析

#### 问题1：状态更新 500 错误

Controller 参数类型改为 `BookingStatus` 枚举后，前端传的是数字（如 `status=1`），Spring 无法自动将字符串 `"1"` 转换为 `BookingStatus.CONFIRMED` 枚举。

#### 问题2：日期选择器偏移

1. 打开日历时 `tempSelected` 为 `null`，点击确定时不会更新值
2. 使用 `toISOString()` 转换日期会受 UTC 时区影响，导致日期偏移（如北京时间 3.5 00:00 转为 UTC 是 3.4 16:00）

#### 1. 添加枚举类型转换器

```java
@Configuration
public class EnumConverterConfig implements WebMvcConfigurer {

    @Override
    public void addFormatters(FormatterRegistry registry) {
        registry.addConverter(new StringToBookingStatusConverter());
    }

    public static class StringToBookingStatusConverter implements Converter<String, BookingStatus> {
        @Override
        public BookingStatus convert(@NonNull String source) {
            try {
                int code = Integer.parseInt(source);
                return BookingStatus.fromCode(code);
            } catch (NumberFormatException e) {
                return BookingStatus.valueOf(source.toUpperCase());
            }
        }
    }
}
```

这样前端传 `status=1` 或 `status=CONFIRMED` 都能正确转换为枚举。

## 第二十四次对话 - 修复 Docker Compose 启动错误

### 用户问题

```
docker-compose up --build -d
error while interpolating services.backend.environment.JWT_SECRET: required variable JWT_SECRET is missing a value
```

### 问题原因

之前使用 `${JWT_SECRET:?...}` 语法强制要求环境变量，但 docker-compose 在某些情况下不会自动加载 `.env` 文件（如文件权限、路径问题等）。

### 解决方案

将 `JWT_SECRET` 改为提供默认值，开发环境开箱即用：

```yaml
# 修改前（强制要求）
JWT_SECRET: ${JWT_SECRET:?请在 .env 文件或环境变量中设置 JWT_SECRET}

# 修改后（提供默认值）
JWT_SECRET: ${JWT_SECRET:-dev-only-secret-key-for-local-development-must-be-at-least-256-bits}
```

### 安全说明

- 开发环境：直接 `docker-compose up --build -d` 即可，使用内置默认密钥
- 生产环境：必须通过 `.env` 文件或环境变量设置强密钥

```bash
# 生产部署
export JWT_SECRET=$(openssl rand -base64 48)
docker-compose up -d
```

| 文件 | 修改内容 |
|------|----------|
| `docker-compose.yml` | JWT_SECRET 改为提供默认值 |
